# 3 需求分析

需求分析是软件工程生命周期中至关重要的环节，它决定了系统开发的最终目标和功能边界。本章将从可行性分析、功能需求、非功能需求等方面对基于Spring Boot+Vue3的在线考试系统进行全面的需求分析。

## 3.1 可行性分析

在进行系统开发前，首先从技术、经济和操作三个维度对在线考试系统进行可行性分析。

（1）**技术可行性**：本系统采用前后端分离架构，后端基于成熟的 Java 21 和 Spring Boot 3 框架，结合 MyBatis-Plus 持久层框架、Spring Security 安全框架进行开发，数据库采用 MySQL 8.4 并配合 Redis 作为缓存以提升性能，同时使用 MinIO 存储考试图片等对象资源，RabbitMQ 处理异步消息。前端采用 Vue 3 (Composition API)、Element-Plus 组件库以及 ECharts 数据可视化库。上述技术栈目前均非常成熟且拥有庞大的开源社区支持，技术方案完全可行。

（2）**经济可行性**：系统所需的技术框架（如 Spring Boot、Vue、MySQL、Redis 等）均为开源免费软件，开发工具（如 IDEA、VS Code）也易于获取。系统部署所需硬件资源要求相对常规，且可以依赖云服务器进行低成本部署，后期维护成本低，具备良好的经济可行性。

（3）**操作可行性**：系统前端采用响应式设计，交互逻辑遵循主流 Web 系统的操作习惯。教师和学生只需通过现代浏览器（如 Chrome、Edge）即可访问系统，无需额外安装客户端插件。系统界面简洁、操作引导清晰，具备良好的操作可行性。

## 3.2 角色分析

系统主要服务于三种用户角色，不同角色具备不同的系统操作权限：

1. **管理员（Admin）**：负责系统的全局管理与基础数据维护。包括用户及权限分配、院系/专业基础数据维护、课程管理、全局监控、数据批量导入及操作审计日志查看等。
2. **教师（Teacher）**：负责教学及考试的核心业务流程。包括教学班管理、题库及题目维护（客观题与主观题）、智能组卷与试卷管理、考试发布、在线监考及反作弊异常处置、主观题阅卷评分，以及考试成绩与学情数据分析。
3. **学生（Student）**：系统的最终使用者。主要功能包括登录系统、查看可参加的考试、进行环境及设备检测、在线答题（具备实时防作弊监测、异常抓拍、切屏限制）、提交试卷以及查看历史考试成绩及错题分析。

## 3.3 功能需求分析

系统根据业务流程划分为多个核心功能模块，主要包括：

### 3.3.1 用户认证与会话安全模块
系统需要提供安全可靠的登录、退出及身份验证功能。通过用户名与密码校验用户身份后签发 JWT 访问令牌，并通过刷新令牌机制维持会话的安全性。同时，支持多角色权限加载、账号状态校验以及限制登录频率等安全措施。

### 3.3.2 基础数据与用户管理模块
主要面向管理员，提供对用户、角色、课程、教学班等系统级基础数据的维护。支持用户信息的增删改查、密码重置、角色分配。在用户角色发生变更时，系统需自动同步更新学生/教师档案及选课关系。支持通过 CSV 文件批量导入用户与教学班，降低管理员的维护成本。

### 3.3.3 教师班级与学生管理模块
面向教师，支持对所授课程对应教学班的管理。教师可查询教学班列表、班级内学生名单，并支持将系统内的学生添加到教学班或从班级中移除。系统需严格校验教师的数据权限。

### 3.3.4 题库与试卷管理模块
- **题库管理**：支持单选、多选、判断、简答等多种题型。提供题目分页查询、新增、修改与删除。题目中的配图通过 MinIO 进行存储管理，后台需具备定时清理无效图片资源的机制。
- **组卷管理**：支持“手动组卷”与“自动抽题组卷”两种模式。自动组卷可根据教师设定的题型、难度、数量进行智能化抽取；系统自动计算试卷总分，且已被考试计划引用的试卷和题目将被限制删除，保障考试数据一致性。

### 3.3.5 考试发布与过程管理模块
- **考试发布**：教师可设定考试的基本信息、考试时间、关联试卷及目标教学班进行考试发布。
- **考试过程**：学生在规定的时间内进入考场答题。系统需在学生答题过程中定时自动缓存答题快照，以防设备断电或网络异常导致数据丢失。支持学生主动交卷与考试结束超时强制自动交卷。高并发交卷时，系统通过 RabbitMQ 异步消费处理提交数据。

### 3.3.6 在线监考与反作弊模块
考试过程需具备多层次的防作弊手段。系统支持配置不同等级的监考策略，学生端实时上报异常事件（如切屏、离开摄像头、多头检测等），并记录异常抓拍证据至 MinIO。教师及管理员可在后台实时监控考试状态，查看单个学生的异常时间线并进行违纪处置。

### 3.3.7 阅卷评分与成绩统计模块
- **阅卷评分**：客观题（选择题、判断题）由系统自动完成批改评分；主观题则进入教师待阅卷列表，系统提供按题目批量阅卷和单份试卷阅卷两种模式，确保评分的准确性和效率。
- **成绩统计分析**：成绩发布后，系统基于 ECharts 自动生成多维度的学情分析图表，包括班级成绩分布情况、最高分/最低分/平均分统计、错题高频排行等，辅助教师评估教学质量。

### 3.3.8 系统监控与操作审计模块
- **考试全局监控**：管理员可实时查看当前系统的考试数量、进行中的考试情况、考生状态分布及异常事件触发数量。
- **操作审计日志**：通过 AOP 切面技术对系统敏感及关键操作（如用户管理、试卷修改、考试发布、阅卷、监考处置等）进行详细记录，包含操作人、IP地址、请求路径、操作结果与时间，保证系统操作的可追溯性。

## 3.4 非功能需求分析

除了满足各项业务功能外，系统还需满足以下非功能性需求，以保障系统高质量运行：

1. **性能需求**：考虑到在线考试场景可能存在的瞬时高并发（如全校统一开考、集中交卷等），系统响应时间应控制在 1 秒以内，交卷接口可承受每秒 500 次以上的并发请求（依赖 RabbitMQ 削峰填谷及 Redis 缓存优化）。
2. **安全性需求**：
   - 接口安全性：通过 Spring Security + JWT 拦截未授权访问，密码采用 BCrypt 强加密算法存储。
   - 数据安全性：提供防 SQL 注入、XSS 跨站脚本攻击防护；涉及考试作弊取证的照片强制使用 MinIO 独立存储。
3. **可靠性需求**：考试过程中允许一定程度的网络断开，学生重新连网后可恢复之前的答题快照进度；系统需具备较强的异常捕获及统一错误处理能力。
4. **易用性与兼容性需求**：前端界面需清爽直观（Vue3 + Element Plus），操作流程符合正常逻辑认知；支持目前主流的 Web 浏览器（Chrome 80+, Edge, Firefox, Safari）。
