一、架构层面

  做得好的地方： ExamService 用 Facade 模式拆分成 17 个专职 Service，职责清晰。Redis 快照 + 定时刷盘的答题保护机制设计合理。防作弊风险评分算法有层次感。

  不足：

  1. 缺乏消息队列解耦 —考试提交后的自动评分、快照刷盘、防作弊事件记录全部是同步调用或定时任务。高并发交卷场景（比如考试结束前最后一分钟几百人同时提交）会造成数据库写入尖峰。建议引入RabbitMQ/Kafka，将提交后的评分、快照持久化、事件统计异步化。
  2. 试卷随机抽题用了 ORDER BY RAND() — 这是 MySQL 性能杀手，题库量大时全表扫描。建议改为先 SELECT id 再应用层随机采样，或者用 TABLESAMPLE。
  3. 缺少缓存策略 — 除了快照用 Redis，试卷内容、题目详情这些高频读低频写的数据没有缓存。一场 200 人的考试，每个人打开试卷都查一次数据库，完全可以缓存试卷快照。
  4. 定时任务单点问题 — ExamAutoSubmitService 和 SnapshotFlushScheduler 用的是 Spring @Scheduled，多实例部署时会重复执行。需要分布式锁（Redisson）或者换成 XXL-Job之类的分布式调度。

---
  二、安全层面

  5. 无接口限流 — 登录接口没有 rate limiting，容易被暴力破解。建议至少对 /api/v1/auth/login 加上滑动窗口限流（比如同 IP 每分钟 10 次）。
  6. Token 存 localStorage — 前端把 JWT 存在 localStorage，任何 XSS 漏洞都能直接偷 token。更安全的做法是 access token 存内存，refresh token 用 HttpOnly cookie。
  7. 硬编码凭证泄露 — LoginView.vue 里 form.username = 'admin'、accountHints 数组包含明文密码，这些会打包进生产构建。应该用 import.meta.env.DEV 条件守卫。
  8. 缺少操作审计日志 — 教师改分、管理员改角色、删除用户这些敏感操作没有审计记录。对于考试系统，成绩相关操作的可追溯性是刚需。
  9. JWT Secret 硬编码在 yaml 里 — 虽然支持环境变量覆盖，但默认值不应该出现在配置文件中。

---
  三、考试体验（学生视角）

  10. 断网恢复能力弱 — 学生考试中如果网络断开，快照无法上传，重连后没有冲突合并机制。建议前端用 IndexedDB 做本地持久化，恢复连接后自动同步，取服务端和本地的最新版本。
  11. 没有断点续考 — schema 里 exam_session 有状态管理，但前端 ExamDoing.vue 没有看到从已有 session恢复答题进度的逻辑。如果浏览器崩溃，学生重新进入应该能恢复到上次快照的状态。
  12. 防作弊误判风险 — 风险评分算法是硬编码的固定权重。一个学生切出去看了一眼微信消息（window blur 2秒），和一个学生持续切屏查答案，得分差异不够大。建议：

    - 权重可配置化，不同考试可以设不同阈值
    - 引入时间衰减，短暂切屏和持续切屏区别对待
    - 给学生一个"申诉"入口
  13. 考试倒计时仅前端 — 如果学生改本地时间或者 JS 被暂停，倒计时不准。关键时间校验应该以服务端 deadline_time 为准，前端定期与服务端对时。

---
  四、教师体验

  14. 主观题批改效率低 — 当前是逐题逐人批改，没有批量操作。如果一道简答题 200 人作答，教师要点 200 次。建议：

    - 支持按题目维度批改（同一题连续看所有学生的答案）
    - 支持批量给分（相似答案一键同分）
    - 支持评分模板/快捷评语
  15. 数据分析维度不够 — 当前 AnalyticsService 提供了分数分布、班级趋势、错题分析，但缺少：

    - 题目区分度和难度系数（经典测量学指标）
    - 学生个人成长曲线（跨考试纵向对比）
    - 知识点掌握热力图
  16. 试卷没有版本管理 — 修改试卷后旧版本直接被覆盖。如果一场考试正在进行中，教师误改了试卷内容，后果不可逆。建议试卷发布后锁定，或者引入版本快照。

---
  五、可靠性与运维
  17. 数据库手动初始化 — spring.sql.init.mode=never，需要手动执行 schema.sql 和 data.sql。虽然有 Flyway，但初始化流程不够自动化。建议 Flyway 管理全部DDL，包括初始数据。
  18. 没有健康检查端点 — 缺少 /actuator/health 或自定义健康检查，无法接入 K8s 探针或监控系统。
  19. 错误处理粒度粗 — GlobalExceptionHandler 存在但前端 HTTP 拦截器对错误的处理比较笼统。401/403 的区分、业务错误码体系可以更完善。
  20. 无前端测试 — 后端有 20 个测试类覆盖核心逻辑，但前端零测试。ExamDoing.vue 这种核心考试页面，至少应该有组件级测试保证快照提交、倒计时、防作弊事件上报的正确性。
